BHUS24-OpROM Backdoor之学习与复现

Created2026-01-29|Updated2026-03-17
|Post View:

BHUS24-OpROM Backdoor

https://github.com/FFRI/orom-backdoor-research

UEFI协议和运行时DXE模块

利用SimpleFileSystemProtocol来读写FAT32/NTFS文件系统，用HttpProtocol来发文件（两个交互接口）

运行时DXE模块在OS加载之后仍然驻留在内存中，可调用Protocol

在Option ROM中植入后门

Option ROM

  1. Option ROM中包含初始化该硬件设备的DXE驱动
  2. 网卡、存储卡、显卡等等都有Option ROM
  3. PCIe初始化在DXE早期执行

优点

  1. 隐蔽性强
  2. 自下而上的攻击模式，不接触用户和内核态

植入场景

  1. 芯片或电路板的生产过程中，恶意代码被直接集成到 SoC 或配套的固件中
  2. 攻击者购买合法的硬件（如网卡或 NVMe 控制器），通过特定工具将恶意代码写入其 OROM，然后通过亚马逊、eBay 等在线平台作为“正品”或“翻新件”售出
  3. 用户态恶意软件写入 OROM
  4. 接触式攻击

三种后门及其特点

Uefi-Only Backdoor

核心在于使用Uefi Protocol进行通信和数据读写

demo演示了通过HttpProtocol和FileProtocol窃取C盘上一个文件

复现

找作者要了源代码，第一个后门需要的条件是目标机器上uefi支持Http协议（IPv4&IPv6），而uefi网络功能目前仅用于PXE 远程启动，但实际上最底层的实现可以直接调用，往目标ip上传数据

uefi setup界面中包含开关http协议栈的功能，比如HP，没有这个功能说明这个机器不支持

编译好的.efi作为uefi应用或者驱动，目前还没搭好局域网的攻击验证环境，先在uefi shell中测试一下主要功能

load命令相当于一次性植入一个驱动，hook的目标点是ExitBootService，相当于在windows logo页面就会执行后续命令

用一个路由器搭了简单的局域网环境

攻击机（我的笔记本）ip：192.168.1.101

靶机ip：192.168.1.103

在uefi shell里面测试连通，互相都能Ping通

但是ping只是基于icmp协议，要用http得用edk2重新编译相关的驱动

然后在uefi shell中先load这些驱动，再运行攻击脚本

在windows logo出现的时候有回显，但没有成功发送数据，日志显示是下面这个函数返回了Access Denied

排查确认了不是防火墙或者Secure Boot的问题

然后看了一下edk2源码，Request函数在两处会返回Access Denied

大概是2种问题：

  1. TxTokens 中已存在相同 Token（并发 / 重入请求）
  2. HTTP 被PCD全局策略禁止

第一处问题，怀疑bootmgfw.efi一类的文件会访问http服务，于是又把这个脚本写成了能在uefi shell中运行的形式

运行结果还是一样，排除Token重用

第二处问题，本质上华硕如果不想让用户使用Http，他们就会把PCD=False写到实际的固件中

ami的代码中只有两处检查了这个PCD选项，而HttpInit的时候没有检查，所以显示Success，但实际上http不可用

在网上找到一篇文章，说编译HttpDxe的时候出现了Access Denied，回去看了edk2源码

默认是FALSE，EDK2官方给的解决方案是改成TRUE，重新编译Http相关驱动，然后在efi shell中把原来的替换掉

drivers命令查看到固件原本加载的HttpDxe驱动，用Unload把自己编译的HttpDxe替换上去

然后重新配置dhcp/ip等等，运行攻击脚本发现数据发送成功，在windows logo有回显

http发送成功，在接收端这边也拿到数据了
