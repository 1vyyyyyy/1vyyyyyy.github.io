Windows内核&Rootkit学习（一）

Created2026-05-07|Updated2026-05-07
|Post View:

Rootkit分类

根据权限级别，Rootkit 通常分为两种基本类型

内核模式 (Kernel Mode) Rootkit

KM Rootkit 以极高权限的用户（NT AUTHORITY\SYSTEM）身份在内核中运行，可以修改内存中的内核结构来操纵操作系统，并躲避杀毒软件等的检测。在 Windows 系统中，这通常意味着它会以内核驱动程序的形式运行。

用户模式 (User Mode) Rootkit

UM Rootkit 是指不包含内核模式组件的 Rootkit。它们利用用户模式技术和 API（例如Hooking和进程注入）来操控操作系统，从而隐藏自身在系统中的存在。UM Rootkit 不符合传统 Rootkit 的定义，因为它们不一定以“root”身份运行（尽管它们可能需要管理员权限才能正常工作），或者以其他超级用户身份运行。如今，许多现代恶意软件家族都包含某种形式的用户模式 Rootkit 组件，因为它们通常会试图逃避杀毒软件和用户自身的检测和清除。

Windows 内核入门

Windows系统架构

与所有现代操作系统一样，Windows 架构分为用户空间和内核空间，每个空间都运行在自己的地址空间中。

用户模式下的每个进程都有一个私有的虚拟地址空间，从 0x00000000 到 0x7FFEFFFF（在 x86 中），或 0x00000000000000000 到 0x00007FFFFFFEFFFF（在 x64 中）

内核模式下的每个进程位于 0x80000000 以上的地址（在 x86 中），或 0xFFFF800000000000 以上的地址（在 x64 中）。

这些地址也可以用符号 MmHighestUserAddress (0x7FFEFFFF) 和 MmSystemRangeStart (0x80000000)来引用。

通常情况下，用户模式下会使用 kernel32.dll 和 ntdll.dll 等操作系统 API 库作为访问操作系统服务的入口点，然后每个操作系统服务都会被转换成一个系统调用，由内核进行处理

设备驱动程序和内核位于 HAL 之上，它们都使用 HAL 的服务来协作，并且位于同一地址空间中

设备驱动程序也是唯一允许用户通过以相同特权级别运行来扩展内核及其功能的机制

kernel32.dll 中API调用示例

为了说明操作系统各层是如何相互叠加的，我们来看一个kernel32.dll API 函数，例如ReadFile

当调用 ReadFile 时，位于 kernel32.dll 中的实现将解析传递给它的参数，并调用 ntdll.dll 中未记录的 NtReadFile

之后，NtReadFile 会将 eax 设置为相应的系统调用号，并执行 SYSENTER 指令（或x64 中的 SYSCALL 指令）

SYSENTER指令会创建一个陷阱，并通过调用存储在MSR 0x176（x86 中为 LSTAR MSR，x64中为KiFastCallEntry ）中的地址（ x64 中为 KiSystemCall64）切换到内核模式。

此函数会保存当前用户模式上下文，并在最终调用相应内核版本的NtReadFile（也称为ZwReadFile）之前设置内核上下文。它位于ntoskrnl.exe中，该程序将完成大部分工作，并调用相应的磁盘驱动程序来执行实际的磁盘读取操作。
