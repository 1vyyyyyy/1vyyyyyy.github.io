brotli库中的整数溢出nday

Created2025-04-04|Updated2026-03-17
|Post View:

在泄露的微星源码中发现使用了v1.0.6的Brotli库

这个库主要是基于 LZ77、哈夫曼编码和字典压缩来节省存储空间，加快数据传输速度

看了一下EDK2里面的MdeModulePkg是引用了这个库，编译了之后是放在DxeIpl这个模块中，用于PEI和DXE阶段间的数据递接

在github上面发现v.1.0.8之前的版本有个CVE-2020-8927，这个源码是22年左右泄露的，说明现在的固件里面如果用了这个，大概率可能没打补丁

这个洞的描述比较模糊，意思是在decoder中传入超过2GiB的数据就会造成整数溢出，这里应该是涉及到了指针的操作

然后去decoder部分找这个洞的补丁，发现有一个地方加了上限值校验

BROTLI_LARGE_MAX_WBITS = 30，即br->avail有可能 > 2^30

找到调用BrotliGetRemainingBytes函数的唯一一个地方，返回值类型被强制转换为int nbytes，然后nbytes被作为BrotliCopyBytes的参数来调用

s->br的类型是BrotliBitReader

然后找到调用CopyUncompressedBlockToOutput的函数BrotliDecoderDecompressStream

其中对br->avail参数的赋值语句有两条

构造：available_in != 0 && br->avail_in != 0 这两个条件

走到下面那个分支，相当于把uint32类型的值赋给了int，如果s->buffer_length超过了int最大值就整数溢出了

这里EDK2写了个uefi app的C文件把Brotli库封装起来了

构造PoC的话从Brostate和AvailableIn两个参数下手

目前有一个未知的问题，社区中说用GCC 11编译brotli子模块会有问题，相当于在这种条件下UEFI固件中DxeIpl驱动中并不包含Brotli库

有可能一些版本的固件中找不到上面的代码，但微星确实没有修
