记一次（真）邪修实验之MSI Bomb Shell

Created2025-10-15|Updated2026-03-17
|Post View:

MSI Bomb Shell —— 你的指针无处遁形

微星自带一个uefi shell的启动项，并且这个shell是被OEM私钥签名之后放到固件里边的，因此在安全启动开启的情况下也可以直接运行，执行任何命令

在Secure Boot开启情况下可执行mm这类命令

微星主板的Secure Boot有两种权限分级

针对某些非 UEFI 或不兼容的硬件/操作系统设置“硬件/操作系统兼容性”并优化功能，
或者设置“最高安全性”以全面验证所有已安装的组件。

比如说体现在，如果是Hardware/OS Compatibility的情况下，uefi shell及其内部命令可执行，文件系统下的脚本也可执行；但是在Max Security的情况下，uefi shell及其内部命令可执行，但是未签名的脚本不可执行

可利用性：

  1. uefi shell自带了内存读写的指令，在任何情况下都可以强制执行
  2. 可以自己构造startup.nsh，从而在进入uefi shell的时候就自动执行

内存中暴露的Protocol Interface指针

CoreStartImage函数用于执行StartImage()函数调用，即运行一个EFI可执行镜像或者application

其中使用到了gSecurity2->FileAuthentication()，作为运行之前的Secure Boot验证步骤

这里之后没有else的分支，因此网上PoC的思路是：将gSecurity2和gSecurity协议指针清零，利用这里没有异常处理代码，可以绕过EFI运行前的验证

gSecurity2这种已注册的协议是通过gBS->LocateProtocol方法在内存中检索指针，获取到指针之后再去访问协议下的函数/方法

写了一个脚本，通过gBS的函数是能直接拿到协议指针地址，以及协议下FileAuthentication函数的地址

在本机上用qemu连windbg，可以看到在启动过程中哪些协议被注册了

圈出来的两个是gSecurity2和gSecurity，他们同时被注册，因此可以先试试把gSecurity2指针清零

uefi shell中提供了一个叫mm的指令，可以直接修改系统内存，不需要任何权限

在运行mm指令后，用dmem检查发现确实把指针修改了

此时再次运行gBS->LocateProtocol(&gEfiSecurity2ArchProtocolGuid, NULL, &gSecurity2)，会被卡死（uefi没有异常处理机制）

我这里尝试运行了一下其他脚本，比如读取另外的protocol，发现也卡死了

查了资料，gBS->LocateProtocol() 的核心逻辑大概如下：

也就是说，它不仅仅是查表，还会：

  • 遍历一个链表；
  • 对链表节点的内容（包括 Guid、Interface 等）进行读取；
  • 其中某些字段（尤其是 Interface）在 DXE 阶段常常是指针指向另一个结构体。

DXE 框架维护的 Protocol Database 是一个双向链表（LIST_ENTRY），每个协议接口实例都挂在 Handle->ProtocolList 上。

在修改之后，实际上破坏了链表结构，导致这个for循环进行不下去了

然后，我写了一个没有用到gBS->LocateProtocol方法的一个efi app（打印系统信息），模拟植入的efi后门程序

重复刚才的操作，将指针清零，发现这里并没有输出结果，并且访问了gSecurity2的指针，然后报错了

在实机上测试，发现UefiInfo还是能够运行，上面的报错应该是qemu模拟器的某种问题

这里我将gSecurity和gSecurity2指针同时在内存中清零，然后通过gBS->LocateProtocol来访问，均返回NULL了

通过Hook验证函数实现Secure Boot绕过

在加载一个efi image或者通过uefi shell 运行efi脚本时，gSecurity2->FileAuthentication 负责对这个文件进行签名校验

edk2的实现如下，下游厂商应该都是沿用的这一套验证机制

既然通过将gSecurity2和gSecurity的指针清零也无法绕过这个判断逻辑（可能LocateProtocol会从其他地方再次获取指针），那就可以考虑从函数本身下手

Secure Boot的验证结果实际上取的是 CoreLoadImageCommon 函数的返回值 Status = SecurityStatus，因此hook的目标就是gSecurity2->FileAuthentication

但是微星主板实际上设计的有一些缺陷：

  1. edk2本来自己写了一个uefi shell的参考实现，大部分下游厂商都把它编译到固件里边了，然后微星在做OEM签名的时候偷懒，把固件里边所有efi都给签名了，就是所谓的”Built in Shell“

  2. 在没设置bios口令的情况下setup界面是可以随意操作的，甚至微星还自己做了一个一键式签名的功能，Secure Boot像是形同虚设

  3. 关于1的问题，一直到今年9月份才有人关注，这个PR之后的EDK2 shell在secure boot开启的情况下，只有签名了才能运行（这个问题技嘉很早自行解决了，但是微星目前还没有发布bios的对应更新，最新的产品应该都能利用）

整体的利用思路是，通过这个签名机制把hook脚本签名，然后进到uefi shell里边load hook，最后可以执行任意efi脚本

不考虑微星设计缺陷的情况下，感觉可以先利用Insyde证书注入的洞，在操作系统里边把hook脚本签名了，然后也能达到一样的效果
