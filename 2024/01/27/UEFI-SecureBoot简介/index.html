Created2024-01-27|Updated2026-03-17

UEFI Secure Boot

UEFI Secure Boot 是内置在UEFI BIOS中的一项安全功能，旨在确保固件环境中仅执行经密钥验证后可信的EFI驱动

UEFI Secure Boot 的执行流程依赖于PK、KEK、db和dbx这四个数据库，这些数据库存储在UEFI变量中，在信任链中被使用

  • PK(Platform Key)：平台密钥PK是UEFI Secure Boot信任链的根，每个平台都有一个对应于制造商的PK，通常直接由制造商预装在固件中。PK私钥部分用于对UEFI固件中的OS Loader和Boot Loader进行签名，生成证书；PK公钥部分用于对这些UEFI驱动的证书进行验签
  • KEK(Key Exchange Key)：密钥交换密钥KEK可以存在多个 ，每一项对应一种可以被启动的操作系统。KEK私钥部分用于对db和dbx、(dbt)进行签名，生成证书；KEK公钥部分用来对 db、dbx、(dbt) 进行验签
  • db(Database)：白名单数据库db是用于对被许可予以加载的 EFI 文件的签名数据库 ，因此 db 的公钥也可以存在很多项
  • dbx(Database Excluded)：黑名单数据库dbx是一个黑名单签名数据库，签名在dbx中的UEFI文件在加载时会被禁止执行，dbx的公钥也可以存在很多项

UEFI Secure Boot 包括两部分，包括对启动镜像的验证和以上四种数据库的验证

其中PK（Platform Key），KEK（Key Exchange Key），db是经验证过的合法映像数据库，dbx是映像黑名单，dbt是时间戳签名数据库

漏洞

CVE-2022-21894

CVE-2022-21894是一种被称为 Baton Drop 的安全漏洞，它涉及到Windows上的 Secure Boot功能绕过，被集成到近两年流行的bootkit–BlackLotus中，从而允许攻击者使用其他危险的BCD选项，例如Bootdebug、Testsigning或 Nointegritychecks。

攻击者需要确保序列化的 Secure Boot策略被分配在一个已知的物理地址之上，默认情况下，它被分配在最低可能的地址，而BCD选项中的avoidlowmemory 元素可以用来确保所有物理内存的分配都在指定的物理地址上。在BlackLotus中，攻击者将保存原始BCD选项的文件替换为了非法构造的bootmgfw.efi，从而允许在不进行完整性检查（nointegritychecks）的情况下加载驱动hvloader.efi ，并加载一个自签名的 mcupdate.dll。

hvload.efi 调用的 BtpLayoutImage函数存在一个回调callback，该回调实际上并没有对存储 Secure 策略的变量 qword_1400E0340进行检查，因此会忽略被删掉的 Secure Boot 策略，直接返回执行加载 .dll 文件的代码，使得攻击者可以利用mcupdate.dll进行提权或植入后门。

CVE-2020-10713 Boot Hole

GRUB2 配置文件是一个文本文件，通常不会像其他文件和可执行文件那样进行签名。该漏洞允许在 GRUB2 中执行任意代码，从而控制操作系统的启动。因此，攻击者可以修改 GRUB2 配置文件的内容，以确保攻击代码在操作系统加载之前运行。通过这种方式，攻击者可以在设备上获得持久性。

GRUB2支持在efi文件中注入签名，参与安全启动的校验流程。但是，由Eclypsium公司发现GRUB2引导程序存在漏洞，导致使用UEFI Secure Boot的系统容易被恶意攻击者利用，进而跳过安全校验流程，修改引导过程或引导的内核及其他恶意动作

  • 经过分析，该漏洞在解析grub.cfg文件时在grub2内部发生了一个缓冲区溢出的漏洞。grub.cfg文件位于EFI分区中，因此只要拥root权限的攻击者，可以跳过修改shim或者grub2引导程序文件的前提下，修改grub.cfg。由于缓冲区溢出使得攻击者可以在UEFI引导阶段获得执行任意代码的权限，从而跳过原有的安全校验流程
  • 当前之所以可以造成解析grub.cfg文件溢出后可以造成攻击者拥有执行任意代码权限的原因是因为：这个缓冲区溢出覆盖了词法分析器Flex在堆中的部分字段，这些字段包括多个内部解析元素，拥有任意读写能力。所以，会造成攻击者获得执行任意代码的权限并篡改引导过程

GRUB2提供的YY_FATAL_ERROR函数并没有停止执行或者退出，仅仅是向串口打印一个错误并返回调用的函数。所以会导致yy_flex_strncpy将长度超大的字符串复制到yytext中从而造成缓冲区溢出。

CVE-2022-4020

CVE-2022-4020影响宏碁Aspire A315-22的某些版本，该漏洞存在于这些宏碁笔记本设备上的HQSwSmiDxe DXE驱动程序中。与联想的问题类似，具有提升权限的攻击者可以利用该漏洞通过修改NVRAM变量来修改UEFI Secure Boot设置。如果NVRAM变量“BootOrderSecureBootDisable”存在，则DXE驱动程序BootOrderDxe只需禁用UEFI Secure Boot。

CVE-2017-7932

ARM架构下的Secure Boot

ARM架构下的 Secure Boot 过程通常涉及打包在多个二进制文件中的本机代码的链式执行，这些二进制文件被加载、验证为真实且未被篡改，并按顺序执行，最终系统准备好按照制造商或用户的预期运行。这通常是通过使用数字签名的二进制文件来完成的，该二进制文件在执行时检查要加载和运行的下一个二进制文件的完整性和真实性，如果验证没有失败，则将控制权转移给它

如果系统在已知的良好状态下打开，并且启动时执行的二进制文件序列被验证为具有有效签名，则该序列的最后一个二进制文件运行后的结果状态被认为是良好且可信的。但是，如果 Secure Boot 过程中执行的二进制文件链中的单个链接无法对序列中的下一个二进制文件执行正确的验证，则系统的结果状态将不再可信。

通常，这条信任链的根是建立在加密密钥材料（签名和加密密钥）和存储在片上ROM中的可信初始固件（bootrom）之上的

高保证启动 (HAB)

高保证启动 (HAB) 是 NXP 在 i.MX 处理器中实施的安全启动。这是内置于片上 ROM 中的功能，负责从引导介质加载初始程序映像，通常是第一阶段引导加载程序

HAB 使用公钥加密技术（特别是 RSA 密钥）来验证启动时执行的映像。图像数据由图像提供者使用私钥离线签名，i.MX 处理器使用相应的公钥验证签名，公钥从要验证的二进制文件的一部分加载

信任链的根烧写在一组称为超级根密钥(SRK) 的 RSA 密钥对上，用于防止加载和使用潜在攻击者提供的任意公钥，对于支持安全启动的 i.MX 处理器，这是一个屏蔽 ROM 和电可编程熔丝 (eFuse)

启动时，HAB bootrom 会从映像的命令序列文件 (CSF)部分加载 SRK 表，计算哈希值并将其与 SRK 熔丝中存储的值进行比较。如果该值匹配，则 Secure Boot 过程将继续验证该映像是否使用正确的密钥进行了正确签名

使用供应商提供的代码签名工具 (CST) 在 X.509 证书中配置公钥。 HAB 功能通过 API 暴露给启动映像代码，而可由命令序列文件中的命令调用。 CSF 包含 HAB 在安全启动期间执行的所有命令、SRK 表以及用于验证要加载和运行的映像的证书和签名

bootrom 的 X.509 证书解析器容易受到基于堆栈的缓冲区溢出的攻击，可以通过加载精心设计的证书来利用该溢出。

安全启动的控制流程如下：

  1. 从存储设备获取中断向量表 (IVT)，或在恢复模式下通过 USB 获取
  2. 执行 DCD 命令
  3. 执行 CSF 命令，这些命令负责安全启动完整性。当设备处于可信模式时，上述步骤 3 中的 CSF 命令通常如下所示：
    1. 安装 SRK 类型的 RSA公钥，其SHA256摘要必须与 SRK 熔丝中写入的相同
    2. 使用 SRK 签名的 X.509 证书安装 CSFK 公钥
    3. 使用 CSFK 验证 CSF
    4. 安装公钥以验证启动映像
    5. 使用之前安装的密钥验证启动映像
  4. 执行下一个引导加载程序阶段

密钥安装是使用 INSTALL_KEY CSF 命令完成的。加载并验证不同的密钥，每个密钥都由先前安装的密钥进行验证（信任根是 SRK 熔丝）。

当安装 X.509 类型的公钥时，函数hab_csf_cmd_install_key（位于地址0xB5C0）将找到负责导入 X.509 证书的插件并连续调用：

  • mod_x509->load_key ( x509_parse_certificate )
  • mod_x509->verify_key ( x509_verify_certificate_signature )

因此，在验证其签名之前会解析整个证书。由于INSTALL_KEY命令可以在任何经过身份验证的命令之前执行，因此攻击者除了能够篡改启动映像或将设备切换到恢复模式之外，无需先决条件即可触发 X.509 解析器中的漏洞

HAB bootrom 使用自己的 ASN.1 和 X.509 解析库，并且由于在解析 X.509 证书扩展时错误调用 ASN.1 库函数而存在漏洞

函数asn1_extract_bit_string（位于地址0xE796 ）可用于使用内部函数asn1_deep_copy_element_data（位于0xEF20 ）复制位字符串对象的内容

递归调用此方法，将 ASN.1 对象的内容复制到作为参数传递的输出缓冲区中。该函数的一个特殊特征是它没有保存输出缓冲区值的输入参数，相反，当NULL指针作为输出缓冲区传递时，所需的大小将返回给调用者，这种行为与 Windows API 非常相似

asn1_extract_bit_string函数应首先使用NULL参数调用，使用malloc分配所需的内存，然后使用新分配的缓冲区再次调用该函数

X.509 规范将keyUsage扩展描述为一个位字符串对象，解析证书扩展时，如果存在字段keyUsage ，则通过指定指向堆栈上的 32 位整数的指针作为输出缓冲区，直接调用函数asn1_extract_bit_string ，keyUsage扩展中大小大于 4 字节的位字符串对象将触发堆栈缓冲区溢出

当函数asn1_extract_bit_string返回时，bootrom代码验证返回的大小不大于4字节，但复制操作已经完成。然后调用函数返回并将其返回地址从堆栈中弹出，这允许攻击者重定向PC寄存器并执行任意代码。
