UEFI入门

Created2024-12-13|Updated2026-03-17
|Post View:

UEFI

UEFI（Unified Extensible Firmware Interface，统一可扩展接口）定义了连接操作系统和硬件体系之间的标准接口，是UEFI Forum发布的一种标准

它是Intel在1998年发起的Intel Boot Initiative 项目的产物，当时由于计算机已经开始使用 x64 架构，同一个时代下的还有Phoenix公司的Core System Software，但后来逐渐被遗忘了。

概念

Boot Manager

启动管理器，它决定UEFI的应用程序和驱动在设备和文件系统中的加载顺序

通过全局变量NVRAM 来管理加载，这个变量可被配置以增删启动项

Image

UEFI规范定义的包含可执行代码的二进制程序文件，包括~应用程序和驱动两种，采用PE32的文件结构

UEFI Service

平台调用接口的集合，这些接口由 UEFI 程序、驱动和 OS Loader 提供，主要涉及 Runtime Services 和 Boot Services

Protocol

代表一种接口，本质上是一种数据结构，包含全局唯一标识符（GUID）、接口数据结构和服务

System Table

用以定位 Protocol ，所有UEFI Image都会接到一个指向 System Table 的指针，通过 System Table 可以访问到固件提供的 Protocol

BCD

即启动配置数据。它作为在ESP引导分区的文件，存储了所有启动相关的配置参数，例如是否可以加载测试签名的程序，是否可以对启动阶段进行调试，引导阶段下一步执行的程序等等。如果我们想要查看或者修改BCD，只需要通过bcdedit.exe工具就可以进行操作。

启动过程

SEC（Security）

这一阶段运行在内存初始化之前，仅CPU的寄存器和CPU内部资源可用

系统会使用临时RAM来进行代码的运行和数据的存取

主要任务：

  1. 处理所有类型的平台重启：非活动状态后上电、从活动状态重新启动、退出深度睡眠模式、各种异常情况
  2. 准备临时内存：一般使用Cache，在ROM上执行的程序依然可用使用C语言（栈）
  3. 提供信任链的根：SEC阶段是整个可信系统的根基要么信任 PI 的其余部分，要么以某种方式检查它们的有效性，通常这里会对PEI加载映像进行HASH校验
  4. 准备必要的数据结构并将其和控制传输到 PEI 阶段：至少传输平台状态、地址和BFV大小、临时 RAM 的地址和大小、地址和堆栈大小

在 x86-64 上SEC 阶段的执行流程：

  1. Reset Vector：重置缓存并切换到ROM中的主要初始化程序（固件入口）
  2. 从实模式切换到保护模式：切换到具有平面内存（不分页）的处理器的保护模式，无需交换
  3. 为BSP初始化MTRR：为不同内存区域缓存已知值
  4. 微代码补丁更新：更新所有可用处理器的微代码。
  5. 初始化NEM：空闲缓存被标记为不可刷新，之后它可以用作临时 RAM，直到主缓存初始化为止，并且它还允许使用堆栈在普通 PL 中编写此初始化，在本例中为C
  6. 早期BSP / AP交互（多处理器相关）：向所有 AP 发送 INIT IPI中断，然后启动 IPI，从所有 AP 接收BIST数据
  7. 切换到PEI入口点：将控制和数据传输到PEI阶段

PEI（Pre-EFI Initialization）

程序从ROM执行转向内存执行，主要任务是初始化足够量的连续RAM，以便为DXE阶段准备执行环境，同时准备并传输有关检测到的设备的数据到DXE阶段

PEI 通常驻留在 SPI 闪存上其自己的固件卷 (FV) 中。它由可执行模块组成，这些模块遵循称为 TE（简洁可执行文件）的文件格式，与 Windows 中众所周知的 PE 格式密切相关。由一个称为PEI Foundation的内核和PEI Module（PEIM）组成

PEI Foundation 负责PEI阶段的服务和流程，对于具有相同架构的处理器和执行特定设备的初始化，还可能存在检查其他模块有效性的信任链支持模块，会建立PEI Services Table，所有的PEI模块都可以访问它

PEIM 可能具有对其他模块的依赖项列表，因此它们的启动顺序不是随机的，而是由 PEI 管理器选择，PEIM 可以填充与位置无关的数据结构 — HOB（Hand-Off Block），其中包含要传递给 DXE 驱动程序的数据以及该驱动程序的 GUID

在 x86-64 上PEI 阶段的执行流程：

  1. 建立“内存”的使用：将数据从 ROM 传输到早期 RAM（Cache作为内存）
  2. PEI Dispatcher：按照从无依赖关系到复杂依赖关系的顺序运行 PEIM 模块，这是一个循环，当没有正在运行的模块时结束
  3. CPI PEIM：初始化CPU，配置MSR等
  4. 平台PEIM：MCH、ICH、平台内置接口（SMBus、Reset等）的早期初始化。使用 SEC 阶段获得的数据确定启动模式（正常、恢复、S3 恢复）。
  5. 内存初始化PEIM：初始化主RAM，并将缓存中的数据传输到主RAM，现在可以正常使用了。该过程取决于上一步中定义的系统状态，例如，当不执行 S3 Resume 内存测试时，这会减少启动时间
  6. 检查是否处于 S3 启动模式：如果系统不在S3 Resume中，则发生HOB和控制到DXE阶段的转移，并且PEI阶段结束；如果仍然位于该位置，则执行 S3 引导脚本的 CPU PEIM，这会将所有处理器返回到其保存的状态
  7. S3 Boot Script Executor：恢复其他设备的状态
  8. 操作系统恢复向量：过渡到操作系统

PEI模块封装了关于处理器、芯片组、设备或平台相关的功能，并由PEI内核负责派遣；各个PEI模块之间的沟通通过PPI来实现（由一个GUID和指针实现）

PEI模块可使用PEI服务 InstallPPI() 和 ReinstallPPI() 来发布一个有效的PPI，其他模块可通过 LocatePPI() 来找到需要的PPI

DXE（Driver Execution Environment）

与 PEI 阶段一样，DXE 阶段也位于其自己的 FV 中。主要区别在于，这次可执行模块不是 TE 文件，而是真正的 PE32 文件。在 64 位机器上，我们应该也能找到 PE32+ 文件，这意味着 DXE 阶段将以 64 位长模式执行

DXE阶段时内存可以正常使用，所有内容的初始化都是根据从 PEI HOB 接收到的信息来执行的

DXE 代码由内核（DXE Foundation）、调度程序和驱动程序组成：内核初始化并启动各种 UEFI 服务：引导服务、运行时服务和 DXE 服务；调度程序负责查找并运行 DXE 驱动程序，这些驱动程序也具有依赖关系，包括负责设置系统管理模式 (SMM)、公开网络、存储和文件系统堆栈，并基本上提供基于 UEFI 的引导加载程序启动内核可能需要的任何服务，从安全角度来看，DXE 阶段特别令人感兴趣，因为它通常是安全启动实施和执行的地方；驱动程序完成硬件初始化并为服务提供硬件抽象。

BDS（Boot Device Selection）

负责发现每个启动设备并执行启动策略，在DXE阶段加载的各个驱动将在此阶段与系统中的硬件进行匹配连接

这些策略也通过全局变量NVRAM来配置，NVRAM会存储操作系统加载器的路径

TSL（Transient System Load）

这一阶段是UEFI将系统控制权转移给操作系统系统前的一个中间时段，其中 OS Loader 作为UEFI在执行

UEFI在此阶段提供 UEFI Shell，当启动服务的 ExitBootServices() 被调用后系统进入RT阶段

RT（Runtime）

在运行时阶段，OS Loader 获得系统的控制权，操作系统内核应该启动并运行，然后它可以继续加载设备驱动程序、生成服务和后台进程等

仅保留RT Services供 OS Loader 和操作系统使用

AL（After Life）

操作系统在关机、待机、休眠或重启处理器后会进入此阶段；在服务器和工作站上，异步事件（SMI中断/NMI中断）的触发也能进入AL阶段

核心服务

Boot Services

顾名思义，Boot Services 用于促进引导过程。从 DXE 阶段开始，直到操作系统加载程序调用 ExitBootServices() 为止，它们都可用。之后，所有 BS 都终止，引导服务内存被回收，剩下的就是Runtime Services。BS 可以进一步分为以下子类别：

  • 虚拟内存服务，支持页粒度的内存管理，此类别中两个最突出的服务无疑是 AllocatePages() 和 FreePages()，类比WinAPI中相关的方法：

  • 池内存服务，支持不分页的小块内存管理。其中包括 AllocatePool() 和 FreePool()，它们可以被认为是 API 对的 UEFI 版本，例如HeapAlloc() / HeapFree()或更熟悉的malloc / free

  • 事件服务，用于同步执行流程，直到发出某个事件信号为止。该类别由 CreateEvent()、CreateEventEx()、NotifyEvent()、SignalEvent()、WaitForEvent()和CloseEvent()等服务组成

  • 协议服务，作为不同 UEFI 模块之间 import 和 export 函数的基础。基本上，UEFI 协议将两件事捆绑在一起：

Runtime Services

与仅在引导过程期间可调用的 Boot Services 不同，即使在符合 UEFI 的 Boot Loader 将控制权传递给操作系统内核之后，运行时服务也会保留在内存中。尽管它们可用，但操作系统没有义务以任何有意义的方式利用它们。例如，Windows 的理念是在运行时限制对 UEFI 服务的访问，而是优先考虑操作系统本机驱动程序，然后提供 ACPI 运行时支持。

与大量可用的启动服务相比，支持的运行时服务列表相对较短，其中包括配置 UEFI NVRAM 变量的服务，即GetVariable()、SetVariable()和QueryVariableInfo()
